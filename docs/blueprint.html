BTD Knowledge Engine

product blueprint — april 2026 — z + max

You collect content from smart people. An interview figures out who you are and what you're trying to do. The system deconstructs that content at the skeletal layer — not what they said, but how they think — and synthesizes it through your personal lens into actionable experiments. It compounds over time.

The value isn't the corpus. The value is borrowing judgment from people who have already made the mistakes. The corpus is a proxy for that judgment when you don't have them in the room. — from the group session, April 2026
Content Layer — What Gets Ingested

Curated sources, deconstructed at the meta layer

Not 10,000 blog posts scraped. 5–20 thinkers, deeply extracted. The content itself is secondary — the patterns underneath are the product.

YouTube / Podcasts
Transcripts via yt-dlp or IQ transcription endpoint. Tagged by creator, date, topics.
Articles / Substacks
Web-clipped via Obsidian Web Clipper. Markdown with frontmatter.
Session Transcripts
Group calls, 1:1s, working sessions. The conversation IS the data.

Storage: files (markdown + frontmatter). Search: LEANN semantic index. No database — the filesystem is the database.

meta-pattern extraction
Extraction — The "Deconstructing Greatness" Layer

Strip to the skeleton. Not what they said — how they think.

Max's insight: don't look at content. Look at the order in which they did something. When did they enter, what did they sequence, what did they skip. The skeletal layer underneath that nobody else is actually looking at.

What gets extracted
Decision patterns. Sequencing instincts. What they skip. When they say "always" vs "it depends." The constraints they check first. The failure modes they warn about. The meta-framework, not the advice.
What doesn't get extracted
Surface-level tips. Tool recommendations (unless the reasoning is captured). "Best practices" without context. Content that only applies to the creator's specific situation.

Stored in wiki/ — an LLM-compiled knowledge base. Concepts, patterns, creator profiles. LLM maintains it; humans rarely touch it directly. LEANN indexes everything for semantic retrieval.

Interview Layer — Where the Product Lives or Dies

The question bank is the IP. Everything else is infrastructure to serve it.

Not a chatbot. A structured elicitation that surfaces what the user actually needs — not what they think they need. The interview is a state machine: each question narrows the solution space, the next question depends on prior answers, some branches terminate early.

The Deviation Mechanic — Same Corpus, Different Output

Two people, same idea. Completely different results.

This is the proof the interview works. If two people show up with "I want to build an AI tool" and get identical output, the questions are too shallow.

Person A
Non-technical · uses Claude Code · learning AI · has 5 hrs/week
System synthesizes
Learning track: start with mental models, not code
Week 1 experiment: describe a system you use daily as a feedback loop
Content: 3Blue1Brown visual explainer, then Meadows "Thinking in Systems" ch.1
Trap flag: don't start with tools — understand the problem shape first
Person B
Technical · built 3 projects · stuck on architecture · launching in 2 weeks
System synthesizes

Output Layer — Structured, Not Prose

The format matters more than the quality of the text

A brilliant insight in a wall of text is worse than a good insight in a structure you can act on. Output is templated and composable, not free-form generation.

For Builders
Architecture doc. File structure. Build order. Trap list. What to build first, what to skip, what the tripwires are.
For Learners
Experiment card. Specific content with timestamps. Exercises. Observable outcomes. Weekly check-in.
For Explorers
Synthesis report. 10 product ideas from your collected content. Cross-industry patterns. "Wake up to a new idea every morning."
compounds over time
Feedback Loop — The Compound Effect

Everything adds up. Nothing streams by.

For the user
Did the experiment work? Did the architecture hold? Did you avoid the trap? Results feed back into the profile. Next experiment is better than the last. Your queries and explorations get filed back into the wiki.
For the system
Questions that stump it → corpus gaps to fill. Popular paths → optimize sequencing. User feedback → better extraction. The wiki gets health-checked: inconsistencies, missing data, stale content.
Where It Breaks — Design Constraints
×
Corpus too shallow → generic answers. Need 5–20 thinkers, deeply extracted. Not breadth — depth.
×
Goal too vague → unfocused protocol. Interview must force specificity before proceeding.
×
No interview → false personalization. Everyone gets the same output. Product is a wrapper.
×
No experiment frame → unfalsifiable habits. "Watch more videos" isn't actionable.
×
No daily surface → forgotten within a week. Must have a re-engagement mechanism.
×
No tracking → no learning, just activity. System must know if the experiment worked.
Technical Stack — Deliberately Minimal

What we're using

  • Storage: Markdown files with frontmatter. The filesystem is the database.
  • Search: LEANN CLI — semantic indexing across all content
  • Transcription: IQ endpoint (OpenAI-compatible, local)
  • Ingestion: yt-dlp for YouTube, Obsidian Web Clipper for articles
  • Wiki compiler: LLM (Claude) — maintains concepts, topics, creator profiles
  • Frontend: Obsidian (view wiki, tracks, visualizations)
  • Interview engine: TBD — state machine, not a chatbot

What we're not using (yet)

  • No database. Files + LEANN. If we need relational queries later, we version up.
  • No web app. This is a tool first. Product second. Validate the question bank before building UI.
  • No graph DB. Might need it for corpus relations later. Resolve by testing retrieval quality on 10 real queries.
  • No fancy RAG. LLM auto-maintains index files and summaries. Works at this scale (~100 articles, ~400K words).
Open Questions — What Max + Z Need to Decide
Q1
Which product are we building? Builder judgment tool (technical people starting projects)? Personalized learning engine (anyone trying to learn/apply ideas)? Borrowed brain for non-technical builders (people using Claude Code who need the judgment without the jargon)? Or all three through the same interview layer?
Q2
Who are the first 5 corpus sources? Karpathy, Ethan Mollick, who else? Systems thinking (Meadows, Senge)? AI builders? Business strategy? The domain determines the audience.
Q3
What's the interview format? Chat-based? Form with adaptive questions? Voice? How long before it feels like a survey vs. enough for real personalization?
Q4
What's the output people actually use? Architecture doc? Experiment card? Daily digest? Product ideas? Need to produce 3 formats manually and get feedback before building anything.
Q5
Does the question bank actually surface different constraints for different people? Only way to know: test it manually with 5 people on the same project prompt. If two people get the same output, the questions are too shallow.
Q6
Build order: Question bank first (validate the IP) → manual testing → interview engine → corpus ingestion → output templates → feedback loop. Or different?
